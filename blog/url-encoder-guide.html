URL Encoding: Why Spaces Become %20 (Complete Guide)

Ever wondered why a space turns into %20 in a web address? This guide breaks down URL encoding — what it is, how it works, which characters are safe, and when you need to encode or decode URLs.

What Is URL Encoding?

URL encoding (also called percent encoding) is the process of converting characters in a URL into a format that can be safely transmitted over the internet. It replaces unsafe or special characters with a "%" followed by two hexadecimal digits that represent the character's ASCII code.

For example, a space character is encoded as %20, because the ASCII value of a space is 32 in decimal, which is 20 in hexadecimal. That is why when you type "hello world" into a search box, the URL becomes ?q=hello%20world.

Quick definition. URL encoding converts non-ASCII, reserved, or special characters into a universally readable format using percent signs and hexadecimal values, ensuring URLs work correctly across all browsers and servers.

Why Do We Need URL Encoding?

URLs were originally designed to only contain a very limited set of characters — basically uppercase and lowercase letters, numbers, and a handful of special characters like dashes and underscores. Everything else needs to be encoded to avoid causing problems.

  • Reliability. Different systems interpret characters differently. Encoding ensures the URL means the same thing everywhere.
  • Safety. Some characters have special meanings in URLs (like ?, =, &, #). Encoding them when they are meant to be literal data prevents parsing errors.
  • Compatibility. Some characters cannot be typed or transmitted in certain contexts. Encoding makes them safe.
  • International characters. Non-English characters like accented letters, Chinese, Japanese, and emoji all need to be encoded to work in URLs.

Which Characters Are URL-Safe?

According to the URL specification (RFC 3986), characters fall into three categories: unreserved (always safe), reserved (have special meaning), and everything else (must be encoded).

Unreserved Characters (Never Need Encoding)

These characters are always safe to use in a URL without encoding:

  • Uppercase letters: A-Z
  • Lowercase letters: a-z
  • Digits: 0-9
  • Special characters: - (hyphen), _ (underscore), . (period), ~ (tilde)

Reserved Characters (Sometimes Need Encoding)

These characters have special meanings in URLs. If you are using them for their intended purpose (like ? to start a query string), leave them as-is. But if they appear in data (like a search query containing an ampersand), they must be encoded:

  • : / ? # [ ] @ – used for the URL scheme, path, fragment, etc.
  • ! $ & ' ( ) * + , ; = – used for query parameters and sub-delimiters
⚠
Common gotcha. The space character is NOT unreserved and NOT reserved — it is simply not allowed in URLs at all. It must always be encoded, either as %20 or sometimes as + in form data (application/x-www-form-urlencoded).

How URL Encoding Actually Works

URL encoding follows a simple, mechanical process. For each character that needs encoding:

  1. Find the character's byte value in UTF-8 encoding
  2. Convert each byte to its two-digit hexadecimal representation
  3. Prepend a percent sign (%) to each pair of hex digits
  4. Use uppercase letters for A-F (the spec prefers uppercase)

Simple Examples

Here are some common encodings you will see all the time:

  • Space → %20 (ASCII 32 = 0x20)
  • ! → %21
  • " → %22
  • # → %23
  • $ → %24
  • % → %25 (meta — encoding the percent sign itself!)
  • & → %26
  • + → %2B
  • / → %2F
  • : → %3A
  • = → %3D
  • ? → %3F
  • @ → %40

Multi-Byte Characters (Emoji and International Text)

For characters outside the basic ASCII range — like accented letters, Chinese characters, or emoji — things get a bit more interesting. These are first encoded as UTF-8 bytes, and then each byte is percent-encoded. This means a single character can turn into multiple %XX pairs.

For example:

  • é (e-acute) → %C3%A9 (2 bytes in UTF-8)
  • € (euro sign) → %E2%82%AC (3 bytes)
  • 😊 (smiling emoji) → %F0%9F%98%8A (4 bytes)
  • 中 (Chinese character) → %E4%B8%AD (3 bytes)

When to Encode and When to Decode

Knowing when to encode and when to decode is half the battle. Here are the most common scenarios:

When You Need to Encode

  • Building query parameters. If you are constructing a URL with user input (like a search query or form data), you must encode the values. Otherwise, a & or = in the input will break your URL structure.
  • Passing URLs as parameters. If you need to send a URL as a query parameter value (like ?redirect=https://example.com), you must encode the inner URL so its special characters do not interfere.
  • Creating URLs with spaces or special characters. Any path segment or filename with spaces, accented characters, etc. needs encoding.
  • Sending data via API requests. When working with REST APIs or form submissions, encoding ensures data integrity.

When You Need to Decode

  • Reading URL parameters. When you receive data from a URL query string, it will be encoded — you need to decode it to get the original text.
  • Debugging URLs. If you see a URL full of % signs and want to understand what it actually says, decoding it makes it human-readable.
  • Processing form data. Server-side code typically needs to decode URL-encoded form data before using it.
  • Analyzing log files. Server logs often contain encoded URLs; decoding them helps with analysis.
Pro tip. Do not double-encode! If you encode something that is already encoded, you will get garbled results. For example, encoding a space once gives %20, but encoding %20 again gives %2520 — which is wrong. Always know whether your data is already encoded before running it through an encoder.

Encoding vs. Decoding: An Example

Say you want to share a link to a Google search for "best coffee shops in New York". Here is how it works:

Original text:
best coffee shops in New York

Encoded URL parameter:
best%20coffee%20shops%20in%20New%20York

Full URL:
https://www.google.com/search?q=best%20coffee%20shops%20in%20New%20York

When you decode that encoded string, you get back the original readable text.

Common URL Encoding Mistakes

1. Encoding the Entire URL

This is probably the #1 mistake. You should never encode an entire URL — only the data parts (query parameter values, path segments that contain special characters). If you encode the whole thing, you break the https://, the slashes, the question mark — everything that makes a URL work.

2. Forgetting to Encode & in Query Parameters

The & character separates query parameters. If a parameter value contains an & and you do not encode it, everything after it will be interpreted as a new parameter. For example, ?q=rock&roll looks like two parameters (q=rock and roll=) instead of one parameter with the value "rock&roll". The correct version is ?q=rock%26roll.

3. Mixing Up + and %20 for Spaces

There are two ways to encode a space: %20 (standard URL encoding) and + (form encoding). The + only works in query strings with the application/x-www-form-urlencoded content type. In the path portion of a URL, + is a literal plus sign, not a space. When in doubt, use %20 — it always works.

4. Double Encoding

As mentioned earlier, encoding something twice corrupts the data. This often happens when data passes through multiple systems that each automatically encode it. Always track whether your data is encoded or not.

5. Encoding Unreserved Characters

While encoding unreserved characters (like letters or numbers) technically still works — the decoder will convert them back correctly — it is unnecessary and makes URLs longer and harder to read. Only encode what needs to be encoded.

URL Encoding in Programming Languages

If you are a developer, you will need to encode and decode URLs in your code. Every major language has built-in functions for this:

JavaScript

JavaScript provides two pairs of functions:

  • encodeURI() / decodeURI() — encodes a full URL, leaving reserved characters alone
  • encodeURIComponent() / decodeURIComponent() — encodes everything except unreserved characters (use for parameter values)

Rule of thumb: Use encodeURIComponent() for individual data values, and encodeURI() for full URLs that are already properly structured.

Python

Python's urllib.parse module has you covered:

  • urllib.parse.quote() — percent-encodes a string
  • urllib.parse.quote_plus() — encodes spaces as + (form-style)
  • urllib.parse.unquote() / unquote_plus() — decodes

PHP

  • urlencode() / urldecode() — form-style encoding (space as +)
  • rawurlencode() / rawurldecode() — RFC 3986 encoding (space as %20)

Frequently Asked Questions

Why is there a %20 in my URL?

%20 is the URL-encoded version of a space character. Whenever you see %20 in a web address, it means there was a space in the original text or filename. Browsers and servers automatically convert spaces to %20 because spaces are not allowed in URLs.

What is the difference between URL encoding and HTML encoding?

They are completely different systems. URL encoding uses %XX format and is for making text safe in web addresses. HTML encoding uses &name; or &#NNN; format (like &amp; for &) and is for making text safe inside HTML documents so it does not get interpreted as HTML tags.

Is + the same as %20?

Sometimes. In query strings submitted with the application/x-www-form-urlencoded format, a + represents a space. But in the path part of a URL, + is literally a plus sign. %20 always represents a space everywhere in a URL. When in doubt, use %20 — it is universally understood.

What does %2F mean in a URL?

%2F is the encoded form of a forward slash (/). You will often see it when a URL is passed as a parameter inside another URL, because the slash would otherwise be interpreted as part of the outer URL's path structure. Decoding %2F gives you back the slash.

Do I need to encode URLs in SEO?

For the most part, modern search engines handle encoded URLs fine. However, having clean, readable URLs with keywords (and without excessive encoding) is better for both users and SEO. If you have a choice between a URL with lots of % characters and a clean one with hyphens, go with the clean one.
