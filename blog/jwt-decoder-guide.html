JWT Decoder: How to Debug JSON Web Tokens

JSON Web Tokens (JWT) are everywhere in modern web authentication, but they can be tricky to debug when something goes wrong. A good JWT decoder lets you inspect token contents, verify signatures, and troubleshoot authentication issues quickly. This guide covers JWT structure, common problems, and how to decode JWT tokens safely.

What Is a JSON Web Token (JWT)?

A JSON Web Token, or JWT (pronounced "jot"), is a compact, URL-safe standard for representing claims securely between two parties. JWTs are widely used for authentication and authorization in web applications, APIs, and microservices.

The key idea behind JWT is that the server can issue a token containing claims about a user — like their user ID, role, and email — and the client can send this token with subsequent requests to prove its identity. The token is digitally signed, so the server can verify that the claims have not been tampered with.

JWT Structure: The Three Parts

A JWT consists of three parts separated by dots: the header, the payload, and the signature. Each part is Base64URL-encoded JSON (with the exception of the signature, which is binary data also encoded in Base64URL).

// JWT format: header.payload.signature
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Header

The header typically contains two fields: the algorithm used for signing (like HS256 or RS256) and the type of token, which is "JWT".

{
  "alg": "HS256",
  "typ": "JWT"
}

Payload

The payload contains the claims — statements about the user and additional metadata. There are three types of claims: registered (standard fields like iss, exp, sub), public, and private claims.

{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022,
  "exp": 1516242622,
  "role": "admin"
}

Signature

The signature is created by taking the encoded header, the encoded payload, a secret key, and running them through the signing algorithm specified in the header. The signature verifies that the token has not been modified in transit.

Important: JWT payloads are not encrypted. Anyone can decode a JWT and read the payload. The signature only proves the token has not been tampered with — it does not hide the data. Never put sensitive information like passwords or credit card numbers in a JWT payload.

Common JWT Claims You Should Know

JWT defines several standard (registered) claims that have specific meanings. Here are the most important ones:

  • iss (Issuer): The entity that issued the token — usually your authentication server.
  • sub (Subject): The subject of the token — typically the user ID.
  • aud (Audience): The intended recipients of the token. Used to prevent tokens from one service being used on another.
  • exp (Expiration Time): The timestamp after which the token is no longer valid. Always check this!
  • nbf (Not Before): The timestamp before which the token must not be accepted.
  • iat (Issued At): The timestamp when the token was issued.
  • jti (JWT ID): A unique identifier for the token, useful for revoking tokens and preventing replay attacks.

How to Use a JWT Decoder

A JWT decoder is an essential tool for any developer working with JSON Web Tokens. Here is how to use one effectively:

  1. Paste the token. Copy your JWT and paste it into a JWT payload decoder. The tool will immediately decode the header and payload sections into human-readable JSON.
  2. Inspect the header. Check the algorithm (alg field). Make sure it is what you expect — HS256 for HMAC, RS256 for RSA, etc. Be suspicious of "none" algorithm tokens.
  3. Examine the payload. Look at all the claims. Is the exp (expiration) time correct? Does the sub (subject) match the expected user? Are there any unexpected claims?
  4. Verify the signature (if possible). If you have the secret key or public key, a good JWT debugger will let you verify the signature to ensure the token has not been tampered with.
  5. Check timestamps. Most JWT decoders will convert Unix timestamps to human-readable dates. Verify that exp is in the future and iat is in the past.

Common JWT Problems and How to Debug Them

Token Expired

This is the most common JWT issue. If the current time is past the exp claim, the token is expired. Check the exp timestamp in your JWT decoder — if it is in the past, your user needs to log in again or use a refresh token.

Invalid Signature

If signature verification fails, the token may have been tampered with, or you might be using the wrong key. Double-check that you are using the correct algorithm and the correct secret or public key. Use a JWT debugger that supports signature verification to test this.

Wrong Algorithm

If your server expects RS256 but the token is signed with HS256 (or vice versa), verification will fail. Always check the alg field in the header and make sure it matches what your server expects. The "none" algorithm should always be rejected by production systems.

Audience Mismatch

If your API checks the aud claim and the token's audience does not match, authentication will fail. This is an important security check to prevent tokens issued for one service from being used on another.

Token Format Issues

JWTs use Base64URL encoding, not standard Base64. If you try to decode a JWT with a standard Base64 decoder, you might get errors. Use a proper JSON web token decoder that handles the URL-safe variant automatically.

JWT Security Best Practices

JWTs are secure when used correctly, but there are many common pitfalls. Follow these best practices:

  1. Always verify the signature. Never trust a JWT payload without verifying the signature first. An attacker can modify the payload if the signature is not checked.
  2. Use short-lived access tokens. If a token is stolen, it can only be used until it expires. Short expiration times (15 minutes to 1 hour) limit the damage window.
  3. Use HTTPS everywhere. JWTs should only be transmitted over HTTPS to prevent interception. Use Secure and HttpOnly cookies for storing tokens in browsers.
  4. Never put secrets in the payload. JWT payloads are Base64-encoded, not encrypted. Anyone can read them. Do not store passwords, API keys, or other sensitive data in JWT claims.
  5. Reject the "none" algorithm. Make sure your JWT library is configured to reject tokens with alg: "none". This is a common attack vector.
  6. Validate all standard claims. Check exp, nbf, iss, and aud on every request. Do not skip these validations — they are important security checks.
  7. Use asymmetric algorithms for distributed systems. RS256 (RSA) or ES256 (ECDSA) let the auth server sign tokens with a private key while services verify with a public key. This is more secure than shared secrets.
  8. Implement proper token revocation. JWTs are stateless by default, but you still need a way to revoke tokens early (for logout, password changes, etc.). Consider a token blocklist or short-lived tokens with refresh tokens.

HS256 vs RS256: Which Algorithm Should You Use?

The two most common JWT signing algorithms are HS256 (HMAC-SHA256) and RS256 (RSA-SHA256). Here is how to choose:

HS256 (HMAC with SHA-256)

HS256 uses a shared secret key to both sign and verify tokens. It is simpler and faster, but everyone who needs to verify tokens also needs the secret key — which means they can also forge tokens. Best for simple systems where only one server issues and verifies tokens.

RS256 (RSA with SHA-256)

RS256 uses asymmetric cryptography: a private key signs tokens, and a public key verifies them. The public key can be safely distributed to any service that needs to verify tokens without giving them the ability to create new ones. Best for microservices, distributed systems, and any scenario where multiple services need to verify tokens.

What to Look for in a Good Online JWT Decoder

Not all JWT tools are created equal. Here is what to look for in a great JWT decoder:

  • Instant decoding. The tool should decode the header and payload as soon as you paste the token, with no button to click.

  • Error messages. If your token is malformed, the tool should give a clear error message explaining what is wrong.

JWT in Different Programming Languages

Every major language has solid JWT libraries. Here are the most popular ones:

  • JavaScript / Node.js: jsonwebtoken, jose
  • Python: PyJWT, python-jose
  • Java: JJWT, Nimbus JOSE + JWT
  • Go: golang-jwt (jwt-go fork)
  • Ruby: ruby-jwt

Always use a well-maintained library instead of implementing JWT yourself. JWT has many subtle security pitfalls that are easy to get wrong.

Frequently Asked Questions

Is it safe to paste my JWT into an online decoder?

It depends on the tool. JWTs contain user data and can sometimes be used to impersonate a user. Only use online JWT decoders that run entirely in your browser and never send your token to a server. zlbox's JWT decoder runs client-side, so your token never leaves your device.

Can anyone decode a JWT?

Yes, anyone can decode the header and payload of a JWT — that is the point. JWT is designed for tamper-evidence, not secrecy. The signature verifies that the token has not been modified, but it does not encrypt the data. Never put sensitive information in a JWT payload.

How do I verify a JWT signature?

To verify a JWT signature, you need the signing key (for HS256) or the public key (for RS256). A JWT debugger with signature verification support will let you paste the key and check if the signature is valid. Always verify signatures in production code — never trust an unverified token.

What is the difference between JWT, JWS, and JWE?

JWT (JSON Web Token) is the general term. JWS (JSON Web Signature) is a signed JWT — the kind you see most often, with three parts. JWE (JSON Web Encryption) is an encrypted JWT that actually hides the payload contents. Most "JWTs" you encounter in practice are JWS tokens.

How long should a JWT last?

Access tokens should be short-lived — typically 15 minutes to 1 hour. Refresh tokens can last longer (days or weeks) but should be revocable. The exact duration depends on your security requirements. Shorter lifetimes reduce the window of opportunity if a token is stolen.

Can JWTs be revoked?

JWTs are stateless by design, so they cannot be revoked in the traditional sense without additional infrastructure. Common approaches include maintaining a token blocklist (checked on every request), using very short-lived access tokens with refresh tokens, or storing token state server-side.
